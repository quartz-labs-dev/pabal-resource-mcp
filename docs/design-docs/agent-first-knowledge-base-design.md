# Design: Agent-first knowledge base structure

## Problem Statement

`pabal-resource-mcp` needs a short `AGENTS.md` that works as a map, not a manual.
The deeper project rules should live in `docs/` so Codex can discover architecture,
tool authoring rules, quality checks, product constraints, and execution plans without
stuffing all guidance into one context-heavy file.

The reference model is OpenAI's Harness Engineering post:
https://openai.com/index/harness-engineering/

The useful takeaway is not "copy their folder tree exactly." The useful takeaway is:
make repository knowledge local, versioned, indexed, and mechanically checkable.

## What Makes This Cool

This turns the repo into something agents can actually operate inside.

Today, an agent can inspect code and infer patterns, but it has to reverse-engineer the
project every time. With a short map plus structured docs, the next run should know:

- where MCP tools are registered
- how schemas, handlers, and docs connect
- what checks must pass before a change is done
- where active plans and completed decisions live
- which rules are stable project invariants, not one-off preferences

That is the whole game. Less prompt archaeology. More useful work.

## Constraints

- Keep `AGENTS.md` short, roughly 100 lines or less.
- Do not duplicate the full README or locale documentation inside `AGENTS.md`.
- Preserve the current package shape: TypeScript, ESM, MCP stdio server, `src/tools/index.ts` registry.
- Use `swpm` by preference for future package operations. Avoid `npm` for new workflow docs except where documenting package consumers.
- Existing public docs under `docs/en-US`, `docs/ko-KR`, and `docs/aso` should remain product/user documentation.
- New agent-operational docs should be distinct from user-facing docs.
- The first version should be useful without adding CI immediately, but should be written so CI/lint checks can be added later.

## Premises

1. `AGENTS.md` should be the table of contents for agents, not the source of every rule.
2. The real system of record should live in `docs/` with stable filenames and cross-links.
3. This repo's highest-value agent knowledge is MCP tool structure, ASO metadata rules, screenshot/icon workflows, and validation commands.
4. Freshness matters. If docs can drift forever, agents will eventually follow stale instructions.
5. The first implementation should create the structure and checklists before adding custom linters.

## Approaches Considered

### Approach A: Minimal map

Summary: Add only `AGENTS.md`, `docs/ARCHITECTURE.md`, `docs/QUALITY.md`, and `docs/PLANS.md`.

Effort: S
Risk: Low

Pros:

- Fastest path.
- Clear improvement over no root agent map.
- Low chance of disrupting existing docs.

Cons:

- Too thin for repeated agent work.
- No dedicated place for active execution plans.
- Quality and freshness rules remain easy to ignore.

Reuses:

- Existing `README.md`
- Existing `docs/en-US`, `docs/ko-KR`, `docs/aso`
- Current `src/tools/index.ts` registry pattern

### Approach B: OpenAI-style knowledge layout

Summary: Add a short `AGENTS.md` plus structured folders for design docs, execution plans, generated references, and agent references.

Effort: M
Risk: Low

Pros:

- Gives agents stable places to look.
- Separates product docs from operational docs.
- Makes future plans and decisions versioned inside the repo.

Cons:

- Still mostly social convention.
- Does not define freshness checks deeply enough.
- Could become another stale docs folder if not maintained.

Reuses:

- Existing docs layout
- Existing TypeScript build and typecheck commands
- Existing MCP tool categories

### Approach C: Agent-first operating structure

Summary: Add the OpenAI-style layout and include explicit freshness rules, tool authoring checklists, quality scoring, and execution plan conventions.

Effort: M
Risk: Medium

Pros:

- Best match for the stated goal.
- Gives Codex concrete operating rules, not vague advice.
- Sets up future CI/lint without requiring it in the first pass.
- Captures human taste once so future agent runs can reuse it.

Cons:

- More files to maintain.
- Needs discipline to avoid writing abstract docs nobody reads.
- If implemented carelessly, it could duplicate README content.

Reuses:

- `src/bin/mcp-server.ts` for server lifecycle rules.
- `src/tools/index.ts` for tool registry rules.
- `src/tools/**` for category-specific examples.
- `docs/aso/ASO_OVERVIEW.md` and `docs/aso/ASO_FIELD_LIMITS.md` for ASO invariants.
- `package.json` scripts for build, typecheck, and test commands.

## Recommended Approach

Choose Approach C.

The implementation should create an agent-operational docs layer without rewriting the
product documentation. The split should be:

- `AGENTS.md`: short map, commands, where to look.
- `docs/ARCHITECTURE.md`: codebase topology, MCP server flow, dependency boundaries.
- `docs/CONTRIBUTING_AGENTS.md`: how agents should add or change tools.
- `docs/QUALITY_SCORE.md`: quality dimensions, current gaps, freshness checklist.
- `docs/PLANS.md`: how execution plans are stored and updated.
- `docs/design-docs/index.md`: index of design docs and decisions.
- `docs/exec-plans/active/`: active implementation plans.
- `docs/exec-plans/completed/`: completed plans with outcomes.
- `docs/generated/`: generated references, such as tool registry snapshots.
- `docs/references/`: external or vendored references useful to agents.

`docs/en-US`, `docs/ko-KR`, and `docs/aso` should remain user and domain docs.

## Proposed AGENTS.md Shape

`AGENTS.md` should contain only:

1. Repository purpose.
2. Must-read docs by task type.
3. Core commands.
4. Non-negotiable rules.
5. Validation expectations.
6. Pointers to deeper docs.

Example task routing:

- Adding or changing an MCP tool: read `docs/CONTRIBUTING_AGENTS.md`, `docs/ARCHITECTURE.md`, then inspect similar files under `src/tools/`.
- Changing ASO behavior: read `docs/aso/ASO_OVERVIEW.md` and `docs/aso/ASO_FIELD_LIMITS.md`.
- Changing screenshot or app icon behavior: read the matching product docs and category implementation under `src/tools/screenshots/` or `src/tools/app-icon/`.
- Planning larger work: create an execution plan under `docs/exec-plans/active/`.
- Finishing larger work: move or copy outcome notes into `docs/exec-plans/completed/` and update `docs/QUALITY_SCORE.md` if quality changed.

## Freshness Rules

The first pass should define checklists, not custom code.

Future CI can enforce these:

- Every public MCP tool in `src/tools/index.ts` appears in generated tool docs or a generated registry snapshot.
- Every tool category has a matching docs entry or an explicit "internal only" note.
- Any change to `src/tools/aso/**` that modifies metadata behavior references `docs/aso/`.
- Any change to `src/bin/mcp-server.ts` updates `docs/ARCHITECTURE.md` if server lifecycle behavior changes.
- Any active plan older than 30 days must be marked stale, completed, or refreshed.
- `AGENTS.md` should not exceed the agreed line budget without moving content into deeper docs.

## Success Criteria

- A new agent can read `AGENTS.md` and know where to inspect next within 60 seconds.
- Adding a new MCP tool has a clear checklist covering schema, handler, registry, docs, and tests.
- Large work has a stable home under `docs/exec-plans/active/`.
- Completed decisions are findable later.
- Product docs and agent-operational docs stop fighting for the same job.
- Future CI checks can be added without inventing a new information structure.

## Distribution Plan

No external distribution is needed for the design itself.

The resulting documentation changes should ship with the package repository and be visible to:

- Codex and other local coding agents.
- Human contributors reading the repo.
- Future CI checks that validate doc freshness.

If this becomes part of contributor workflow, the docs should be included in normal pull requests and reviewed like code.
